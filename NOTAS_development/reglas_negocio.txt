Implementación en tu lógica de Packs

Aquí te preparé una función de "Sentido Común" que calcula las 10 fechas del pack, saltándose feriados y fines de semana no deseados:

import holidays
from datetime import timedelta

def generar_fechas_pack(fecha_inicio, dias_semana, cantidad_clases=10):
    """
    dias_semana: lista de enteros [0, 2, 4] para L, M, V (ISO weekday)
    """
    feriados_cl = holidays.Chile(years=[2025, 2026])
    fechas_finales = []
    fecha_actual = fecha_inicio

    while len(fechas_finales) < cantidad_clases:
        # 1. ¿Es uno de los días elegidos?
        if fecha_actual.weekday() in dias_semana:
            # 2. ¿Es feriado en Chile?
            if fecha_actual not in feriados_cl:
                fechas_finales.append(fecha_actual)
            else:
                # El "sentido común": Si es feriado, no contamos la clase y saltamos
                print(f"Saltando feriado: {fecha_actual} - {feriados_cl.get(fecha_actual)}")
        
        fecha_actual += timedelta(days=1)
        
    return fechas_finales

2. El flujo de "Congelamiento" (Business Logic)

Mencionaste que por enfermedad grave o accidentes se pueden congelar las clases. Esto es un reto para el calendario.

Cómo manejarlo en Django:
No borres las reservas. Agrega un campo estado (choices) a tu modelo Reserva.

    Si la alumna congela, marcas las reservas futuras como FROZEN.

    Esto libera el cupo en el Reformer (capacidad vuelve a subir).

    Cuando la alumna vuelve, el sistema busca las próximas sesiones disponibles para "re-agendar" esas clases congeladas.

3. ¿Por qué no usar una API externa?

Existen APIs como "Feriados de Chile", pero para una lógica de negocio core, la librería local es mejor por:

    Velocidad: El cálculo es instantáneo al guardar el pack.

    Disponibilidad: Si la API externa se cae o cambia su formato (JSON), tu proceso de ventas se detiene.

    Predictibilidad: Puedes testear fechas de 2027 o 2028 hoy mismo sin depender de internet.

2. Lógica de Precios Autogestionada (Business Logic)

Mencionaste que el alumno puede autogestionar un "Pack Reducido" (2 a 9 clases) a $20.000 c/u.
El sentido común en Django dicta que no debes dejar que el frontend envíe el precio (sería hackeable). El backend debe calcularlo basado en la cantidad de fechas seleccionadas.
Implementación del "Price Manager" en el Modelo
Python

from django.core.exceptions import ValidationError

class Suscripcion(models.Model):
    TIPO_CLASE = [
        ('PACK_10', 'Pack 10 Clases ($140.000)'),
        ('REDUCIDO', 'Pack Reducido ($20.000 c/u)'),
        ('SUELTA', 'Clase Suelta ($25.000)'),
        ('PRUEBA', 'Clase de Prueba Sábado ($15.000)'),
    ]
    
    usuario = models.ForeignKey('auth.User', on_delete=models.CASCADE)
    tipo = models.CharField(max_length=20, choices=TIPO_CLASE)
    cantidad_clases = models.PositiveIntegerField()
    total_a_pagar = models.DecimalField(max_digits=10, decimal_places=0, editable=False)

    def save(self, *args, **kwargs):
        # Lógica de Precios (Sentido Común)
        if self.tipo == 'PACK_10':
            if self.cantidad_clases != 10:
                raise ValidationError("El Pack 10 debe tener exactamente 10 clases.")
            self.total_a_pagar = 140000
            
        elif self.tipo == 'REDUCIDO':
            if not (2 <= self.cantidad_clases <= 9):
                raise ValidationError("El Pack Reducido es de 2 a 9 clases.")
            self.total_a_pagar = self.cantidad_clases * 20000
            
        elif self.tipo == 'SUELTA':
            self.total_a_pagar = 25000
            
        elif self.tipo == 'PRUEBA':
            self.total_a_pagar = 15000
            
        super().save(*args, **kwargs)
